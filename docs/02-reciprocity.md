# 02. 호혜 시스템 — 품앗이 기반 무임승차·체리피커 제재

품앗이의 원리: **먼저 받은 품은 반드시 갚는다.** 이를 기계가 집행 가능한 규칙으로 변환한다.
전통 개념 매핑 — 품앗이(호혜 원장), 두레(공동 의무 쿼터), 계(상호 보증·예치).

## 제재 대상 행동 유형

| 유형 | 행동 | 주 대응 장치 |
|---|---|---|
| 소비형 | 자기 RFC에 남의 자원만 쓰고 기여 안 함 | 호혜 원장 (§1) |
| 선별형 | 쉽고 보상 큰 티켓만 골라감 | 두레 쿼터 + 동적 현상금 (§2, §3) |
| 먹튀형 | 자원 약정만 하고 이행 안 함 | 예치금 + 약정 이행 평판 (§3) |
| 저품질형 | 대충 만든 결과를 다수 투척해 당첨 노림 | 예치금 몰수 + 평판 (§3) |
| 도장형 | 검증 없이 승인만 찍는 검증자 | 허니팟 (§4) |

## §1. 이중 원장 — 평판과 호혜의 분리

- **평판 원장**: 일을 *잘하는가* (역할별 정확도, [01-governance.md](01-governance.md) 참조)
- **호혜 원장**: 받은 만큼 *갚는가* (본 문서)

```
크레딧 적립 = 검증 통과한 작업량 × 난이도 계수 × 희소성 계수
크레딧 차감 = 내 RFC/티켓이 소비한 타 노드의 작업량
호혜 지수  = (최근 90일 기여) / (최근 90일 소비)
```

- 호혜 지수가 임계치 아래 → 제안의 티켓 발급 우선순위 하락, 투표 가중치 축소. **차단이 아니라 감속.**
- 크레딧 **감가(decay)** 적용: 과거 적립분으로 장기 무임승차 불가. 품은 지속적으로 갚아야 한다.
- 크레딧은 원칙적으로 양도 불가 (매집·매매로 인한 금융화 방지).

## §2. 두레 쿼터 — 의무 배분

공동 작업은 선택이 아니다.

- 개발 티켓 N건 수행 자격 = 검증·재현 등 공익 임무 일정 비율 수행 (예: 개발 3 : 검증 1 : 재현 1)
- **작업 선택 엔트로피 감시**: 네트워크 평균 대비 특정 난이도·보상 구간에만 편중된 노드는
  선별형으로 판정, 매칭 우선순위 하락
- **검증 무작위 지명**: 검증 임무는 노드가 고르지 않고 능력 가중 추첨으로 배정.
  지명 거부 누적 시 호혜 지수 차감. (선택제로 두면 어려운 검증만 공석이 된다)

## §3. 예치금과 동적 현상금 — 시장이 절반을 해결

- **예치(escrow)**: 티켓 클레임 시 크레딧 일부 잠금. 기한 내 미제출·저품질 제출 시 일부 몰수.
  먹튀형·저품질 스팸을 동시에 억제.
- **동적 현상금**: 장기 미클레임 티켓은 보상 상승, 인기 티켓은 하락.
  선별형 체리피킹의 상당 부분은 제재가 아니라 **가격이 교정**한다 —
  쉬운 일만 고르는 것이 경제적으로 손해가 되도록.

## §4. 허니팟 — 도장형 검증자 탐지

검증이 무너지면 시스템 전체 신뢰가 무너진다. 가장 강한 방어가 필요하다.

- 플랫폼이 **합성 저장소·격리된 평가 세트의 의도적 결함 패치(카나리)**를 검증 큐에 주기적으로 혼입
- 카나리를 승인한 검증자: 검증 평판 대폭 삭감 + 일정 기간 검증 자격 정지
- 보조 신호: 다수 의견과의 100% 일치율 + 비정상적으로 짧은 판정 시간 → 도장형 후보 표기
- 실제 외부 프로젝트에 결함을 제출하지 않으며 카나리는 병합 불가능한 격리 환경에서만 사용
- **허니팟 생성 규칙과 투입 스케줄은 비공개 평가 로직**
  ([03-architecture-github.md](03-architecture-github.md)의 중앙 잔존 항목)

## §5. 점진적 제재 — Ostrom 원칙

오스트롬의 공유자원 관리 원칙 중 "점진적 제재(graduated sanctions)"를 채택한다.
즉시 추방형은 오판 한 번에 선량한 노드를 잃고, 복귀 경로가 없으면 위장 재가입(Sybil) 유인만 커진다.

```
1단계  경고 + 매칭 우선순위 하락
2단계  크레딧 적립률 50% 감액
3단계  보호관찰: 저위험 티켓(문서·테스트)만 배정
4단계  격리: 소비 불가, 기여만 가능 — "빚진 품을 갚아야 복귀"
5단계  추방 (반복적·악의적 행위 한정)
```

4단계가 품앗이의 핵심이다: 추방이 아니라 **갚을 기회**.

Ostrom 8원칙 매핑: 명확한 경계(옵트인 레지스트리·노드 신원), 현지 조건 부합 규칙(프로젝트별 정책),
집합적 선택(RFC 투표), 감시(상호 검증·허니팟), 점진적 제재(본 절), 저비용 분쟁 해결(이의제기 티켓),
자치 인정(Minority Fork), 중층 구조(프로젝트→연합).

## §6. 계(契)식 보증인 제도 — 신규 노드와 Sybil 방어

- 신규 노드는 보호관찰로 시작: 낮은 크레딧 한도, 투표권 없음, 소규모 검증 작업으로 실적 축적
- 기존 노드가 **보증**하면 한도 상승. 단, 피보증 노드가 제재받으면 **보증인의 호혜 지수도 연대 차감**
- 효과: 가짜 노드 대량 생성 비용을 "기존 평판의 위험 부담"으로 전환
- 1차 Sybil 필터: GitHub 계정 연령·활동 이력 + 서명 커밋

## 열린 결정 (파라미터 튜닝 필요)

초기값은 가설이며 운영 데이터로 조정한다. 사용자(설계자)가 정해야 할 값:

- 호혜 지수 임계치 (초안: 0.6) 와 측정 창 (초안: 90일)
- 두레 쿼터 비율 (초안: 개발 3 : 검증 1 : 재현 1)
- 크레딧 감가율 (초안: 월 10%)
- 예치 비율과 몰수 비율
- 허니팟 혼입률 (검증 큐의 몇 %인가 — 높으면 낭비, 낮으면 탐지 실패)
- 보증인 연대 차감 비율
